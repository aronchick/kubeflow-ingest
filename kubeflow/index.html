Pipeline Examples

Here's how it works. These pipelines poll enterprise sources, transform documents, and land them on a PVC for Kubeflow to process.

SharePoint ‚Üí PVC ‚Üí KFP

Poll SharePoint for document changes. Route PDFs and DOCX through Dockling for text extraction. Write to PVC for Kubeflow to pick up.

expanso-sharepoint-ingest.yaml ~45 lines 
# ============================================================
# SharePoint ‚Üí PVC: Enterprise Document Ingestion
# ============================================================
# Poll SharePoint for new/updated documents
# Route PDF/DOCX through Dockling for text extraction
# Write to PVC for Kubeflow Pipeline to process
# ============================================================

input:
  generate:
    interval: 60s  # Poll every minute
    mapping: |
      # Simulated SharePoint document metadata
      # In production: use http_client with Microsoft Graph API
      root.id = uuid_v4()
      root.filename = "HR_Policy_" + now().format_timestamp("2006-01-02") + ".pdf"
      root.content = "Annual leave policy updated to 25 days..."
      root.source = "SharePoint"
      root.site = "HR Department"
      root.modified_at = now()

pipeline:
  processors:
    # Log incoming document
    - log:
        level: INFO
        message: 'New document: ${! this.filename } from ${! this.source }'

    # Route PDF/DOCX through Dockling for text extraction
    - switch:
      - check: this.filename.has_suffix(".pdf") || this.filename.has_suffix(".docx")
        processors:
          - log:
              message: 'Routing ${! this.filename } through Dockling...'
          - subprocess:
              name: ./mock_dockling.sh
              args: ["extract"]
          - mapping: |
              root = this
              root.transformed = true
              root.transformer = "dockling"

    # Add metadata for KFP
    - mapping: |
        root.kfp_metadata = {
          "ingested_at": now(),
          "source_system": this.source,
          "document_id": this.id,
          "ready_for_embedding": true
        }

output:
  file:
    path: /mnt/pvc/incoming/${! this.id }.json
    codec: lines

Confluence ‚Üí PVC ‚Üí KFP

Watch Confluence spaces for page updates. Extract content and metadata. Land on PVC with KFP-ready structure.

expanso-confluence-ingest.yaml ~40 lines 
# ============================================================
# Confluence ‚Üí PVC: Wiki Content Ingestion
# ============================================================

input:
  generate:
    interval: 120s  # Poll every 2 minutes
    mapping: |
      # Simulated Confluence page metadata
      # In production: use http_client with Confluence REST API
      root.id = uuid_v4()
      root.title = "Engineering Runbook: Incident Response"
      root.space_key = "ENG"
      root.content = "When an incident occurs, follow these steps..."
      root.source = "Confluence"
      root.last_modified = now()
      root.author = "Platform Team"

pipeline:
  processors:
    - log:
        level: INFO
        message: 'Confluence page updated: ${! this.title }'

    # Clean HTML content (Confluence returns HTML)
    - mapping: |
        root = this
        # Strip HTML tags for plain text
        root.plain_text = this.content.re_replace_all("<[^>]*>", "")
        root.word_count = this.plain_text.split(" ").length()

    # Structure for KFP
    - mapping: |
        root.kfp_metadata = {
          "ingested_at": now(),
          "source_system": "confluence",
          "space": this.space_key,
          "document_id": this.id,
          "ready_for_embedding": true
        }

output:
  file:
    path: /mnt/pvc/incoming/${! this.id }.json
    codec: lines

Bonus: Trigger KFP on New Documents

Watch the PVC for new documents and automatically trigger a Kubeflow Pipeline run.

expanso-kfp-trigger.yaml ~25 lines 
# ============================================================
# Watch PVC ‚Üí Trigger KFP Pipeline
# ============================================================

input:
  file:
    paths: ["/mnt/pvc/incoming/*.json"]
    scanner:
      to_the_end: {}

pipeline:
  processors:
    - mapping: |
        root.pipeline_name = "document-embedding-pipeline"
        root.params = {
          "input_path": "/mnt/pvc/incoming/" + this.kfp_metadata.document_id + ".json",
          "source_system": this.kfp_metadata.source_system
        }

output:
  http_client:
    url: "${KFP_API_URL}/apis/v2beta1/runs"
    verb: POST
    headers:
      Content-Type: application/json

FAQ

How does this integrate with existing OpenShift AI?

Expanso runs as a pod in your OpenShift cluster. It writes to a PVC that your Kubeflow Pipelines already know how to read. No changes to your existing ML infrastructure.

What about authentication to SharePoint/Confluence?

Expanso supports OAuth 2.0 flows through the http_client input. You configure credentials via environment variables or Kubernetes secrets. The pipeline handles token refresh automatically.

Can I use this with Feast instead of raw files?

Absolutely. Change the output to http_client pointing at your Feast materialization endpoint, or write directly to the feature store's backing database.

What about schema detection?

Expanso can infer schema from incoming documents and validate against expected formats. Use the json_schema processor for validation, or transform documents into your standard format with mapping.

How do I monitor pipeline health?

Expanso exposes Prometheus metrics out of the box. You'll see message counts, processing times, error rates, and more. Integrates with your existing observability stack.
